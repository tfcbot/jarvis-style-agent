# PRD — Jarvis-Style Agent

## Problem

People want a "Jarvis": a thing they can talk to out loud that talks back and actually does things for
them. The pieces to build one exist (voice models, agent frameworks, cheap hosting), but wiring them
together correctly is fiddly and easy to get subtly wrong. The hard parts are not obvious: keeping
keys off the client, making a stateless voice loop feel live, getting an agent backend to deploy and
stay on, making the whole thing the buyer's own rather than a rental.

Most people building this either stall on the plumbing or end up with a brittle demo that only runs on
their laptop. They do not need another hosted product to rent. They need the **foundation** built
correctly, once, in their own repo, so they can build on it forever.

## What this is

A **build guide** handed to a coding agent. The agent reads it and builds the buyer a working
foundation: a particle-orb voice frontend, an agent backend, deployed to Vercel, where the buyer can
say "hello" and hear a spoken reply. The guide ships instructions and verbatim-pinned code, not a
runtime. The buyer owns everything the agent produces.

## Who it is for

| User                       | Job to be done                                                            |
| -------------------------- | ------------------------------------------------------------------------- |
| Builder with a coding agent | "Give me the clean, correct base for a voice agent so I can build on it."  |
| Founder / operator          | "I want my own talking agent I control, not a subscription to someone's."  |
| Tinkerer / learner          | "Teach me the real architecture (frontend/backend/context/memory + BFF)."  |

The common thread: they have (or can run) a coding agent, and they want to **own** the result.

## The four concepts (the product's spine)

The guide is organized around four named layers and one pattern. This is deliberate: once the buyer's
agent understands these, it can build anything on the foundation.

- **Frontend** — what the person sees and talks to (the orb; renders, captures voice, plays speech).
- **Backend** — the agent brain (takes a message, thinks, replies; where intelligence and secrets live).
- **Context** — what the agent knows during a conversation (instructions + live transcript).
- **Memory** — what the agent knows across conversations (durable facts that outlive a session).
- **Backend-for-Frontend (BFF)** — the pattern: the frontend talks only to one backend the buyer owns;
  that backend talks to everything else. One home per integration, secrets server-side.

## Goals

- A buyer's agent can go from this guide to a deployed, talking orb with no human help beyond browser
  sign-ins and pasting keys.
- The buyer owns 100% of the result: their repo, their Vercel, their keys.
- Every hard-to-get-right piece is pinned verbatim so the build is reproducible, not guessed.
- The foundation is extensible by design: adding tools, memory, or new surfaces is a documented,
  low-friction next step, not a rebuild.

## Non-goals

- **No hosted runtime.** This guide ships instructions; the buyer runs everything. There is no service
  to operate on their behalf and no per-buyer support burden baked into the price.
- **No pre-built integrations.** Shopify, analytics, CRMs, etc. are examples in the extensibility guide,
  not part of the foundation. The agent builds what the buyer asks for, when they ask.
- **No mobile / native app.** The orb is a web app.
- **Not a model or a voice vendor.** The guide uses the Vercel AI Gateway (keyless) for model + voice
  by default and documents swaps; it does not lock the buyer to any provider.

## Acceptance — the foundation is "done" when

The buyer opens the deployed orb in a browser, taps the mic, says **"hello,"** and **hears a spoken
reply** generated by their own agent brain. Specifically:

1. The **orb frontend** is deployed to Vercel and loads.
2. The **EVE backend** is deployed to Vercel and answers `POST /v1/chat/completions` for a valid bearer.
3. Voice round-trips: speech in (STT) → brain → speech out (TTS), all through the buyer's own BFF routes.
4. **No secret reaches the browser.** Model key, gateway key, and shim secret all stay server-side.

`guide/06-verify.md` is the exact checklist. When it passes, the foundation is complete.

## Where the foundation ends, your build begins

This is the value, not a limit. The guide nails the foundation, the part that is easy to get wrong, so
everything after it is easy. Past "hello," the same agent builds the buyer's backend out cleanly:

- Add **tools** the agent can call (read data, take actions) — `guide/07-extensibility.md`.
- Give it **memory** so it remembers across sessions — `guide/04-memory.md` (Cognee, bring-your-own-key).
- Shape its **context** and persona — `guide/03-context.md`.
- Expose it to other apps as an **MCP server**, swap the **backend** (EVE → a minimal Hono API), or
  swap the **voice** (gateway → ElevenLabs) — all in the extensibility guide.

The foundation is the clean base. What gets built on it is the buyer's to decide, and the guide hands
their agent the playbook to build it cleanly.
